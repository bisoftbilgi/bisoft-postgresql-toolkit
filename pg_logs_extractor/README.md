## Overview

`pg_logs_extractor` is a high-performance, asynchronous Rust-based pipeline that reads PostgreSQL log files, classifies events, parses multiline outputs (such as `auto_explain` execution plans), and writes the normalized results directly into a centralized PostgreSQL database. 

It handles complex routing for `ERROR/FATAL` logs, `AUDIT` logs, connection events, and intelligently extracts millisecond durations from execution plans.

## Architecture Update: Logstash to Rust

Previously, this pipeline utilized a Logstash/JVM architecture. However, capturing sub-millisecond query execution plans (`auto_explain.log_min_duration: '1ms'`) generates massive amounts of multiline logs. Logstash's JVM footprint caused significant RAM overhead (1-2 GB). 

This project has been completely rewritten in **Rust** (using `tokio` and `sqlx`). 
**The result:** Memory consumption has dropped to **~10-15 MB**, parsing speed has exponentially increased, and the system now features zero-downtime dynamic file discovery.

## Features

- **High-Performance Ingestion:** Tail and process rotating logs with minimal CPU/RAM footprint.
- **Smart Split-Routing:** Automatically categorizes and routes logs into specific tables (`error_logs`, `auto_explain_logs`, `connection_logs`, `audit_logs`).
- **Multiline Parsing:** Intelligently buffers and parses multiline query execution plans generated by `auto_explain` without breaking the log structure.
- **Auto-Migration:** Automatically executes DDL statements to create required tables on the destination database at startup.
- **Duration Extraction:** Uses native Regex to extract the exact execution time (`duration: X ms`) from explain plans as numeric data.
- **Dynamic File Discovery:** Uses `glob` patterns to track new log files automatically without service restarts.

## How It Works

1. **Dynamic File Tracking:** The service scans the path defined in `config.toml` every 10 seconds to discover new `.log` files.
2. **Asynchronous Tailing:** Each discovered file is tailed in its own asynchronous `tokio` thread.
3. **Regex & Multiline Buffering:** Log lines starting with a timestamp trigger a new event. Subsequent lines (like SQL queries or EXPLAIN plans) are buffered into a single multiline string.
4. **Classification:**
   - Contains `duration:` and `plan:` -> Routed to `auto_explain_logs`.
   - Level is `ERROR`, `FATAL`, or `PANIC` -> Routed to `error_logs`.
5. **Database Insertion:** Parsed data is pushed via asynchronous channels to a connection pool (`sqlx`) and inserted into the centralized DB.

Prerequisites
Rust Toolchain: Cargo and Rustc installed on the host/build machine.

Destination PostgreSQL: Reachable from the host running the extractor.

Log File Permissions: PostgreSQL log files must be readable by the user executing the Rust binary.

PostgreSQL Log Source Setup
Ensure your source PostgreSQL instance is configured to log necessary events. In your postgresql.conf:

Kod snippet'i
logging_collector = on
log_directory = 'log'
log_filename = 'postgresql-%a.log'
log_line_prefix = '%m [%p] user=%u,db=%d, client_ip=%h app=%a '
shared_preload_libraries = 'auto_explain,pgaudit'
pgaudit.log = 'all'
auto_explain.log_min_duration = '1ms'
auto_explain.log_analyze = on
Destination PostgreSQL Setup
The Rust service features Auto-Migration. You do not need to manually run DDL scripts. On startup, pg_logs_extractor will automatically verify and create the following tables if they do not exist:

connection_logs

audit_logs

auto_explain_logs

error_logs

Configuration (config.toml)
Create or modify the config.toml file in the root directory. This replaces the old .env and servers.yml files, centralizing all settings.

Ini, TOML
# Output destination configuration
output_mode = "postgresql"

# Centralized DB Connection String
db_url = "postgres://postgres:YourPasswordHere@127.0.0.1:5433/postgres"

# Metadata for the source server
hostname = "bfm01"

# Glob pattern to discover logs
log_pattern = "/var/log/postgresql/postgresql*.log"

# Batch processing interval
flush_interval_secs = 2
Run & Monitor
1. Build the binary (Release mode for optimized performance):

Bash
cargo build --release
2. Run the service:

Bash
cargo run --release
(Alternatively, you can run the compiled binary directly or set it up as a systemd service: ./target/release/pg_logs_extractor)

Verification (SQL Checks)
Run the following queries on your destination PostgreSQL to verify data ingestion:

SQL
SELECT * FROM auto_explain_logs ORDER BY id DESC LIMIT 5;
SELECT * FROM error_logs ORDER BY id DESC LIMIT 5;
Troubleshooting
No logs are being processed
Ensure the log_pattern in config.toml matches the actual log directory.

Verify that the user running the Rust application has read permissions for /var/log/postgresql/ (e.g., sudo chmod 644 /var/log/postgresql/*.log).

Database connection failures
Check the db_url string in config.toml. Ensure the port, username, password, and database name are correct.

Ensure the destination PostgreSQL allows connections from the host IP (pg_hba.conf).

High CPU usage
Ensure you compiled the application using the --release flag. Debug builds are significantly slower and consume more resources.

Security Notes
Protect config.toml: It contains plain-text database credentials. Ensure it has strict file permissions (chmod 600 config.toml).

Do not commit secrets: Ensure config.toml is included in your .gitignore if it contains production credentials.

Principle of Least Privilege: The PostgreSQL user defined in db_url only needs INSERT and CREATE TABLE privileges.
